# Minio

**Official Website:** [https://min.io/](https://min.io/)

## Purpose in Architecture

`Minio` is a **high-performance object storage server compatible with the Amazon S3 API**. Its role in our ecosystem is to provide private, secure, and reliable object storage within our Kubernetes cluster.

It is a fundamental piece for various tasks, including:
-   Serving as the backup target for [Longhorn](./longhorn.md).
-   Storing artifacts, logs, or any other unstructured data generated by our applications.
-   Providing a "data lake" for AI or data analysis applications.

## Basic Operation

-   **Object Storage:** Unlike block storage (like Longhorn), Minio stores data as "objects" in "buckets." Each object consists of the data itself, metadata, and a unique identifier.
-   **S3 Compatibility:** Its biggest advantage is that it is fully compatible with the S3 API. Any tool or SDK (like Python's `boto3` or `aws-cli`) that works with AWS S3 will work with Minio, simply by changing the endpoint URL.
-   **User Interface:** It includes a very intuitive web console for creating buckets, managing access policies, and browsing stored objects.

## Project Integration

-   **Base:** The configuration for deploying Minio is located in `k8s/base/apps/minio/`.
-   **Persistent Storage:** Minio needs persistent storage to save the objects. It uses a `PersistentVolumeClaim` that can be satisfied by [Longhorn](./longhorn.md) or any other available block storage solution in the cluster.
-   **Access:**
    -   **Internal:** Other applications within the cluster (like Longhorn for its backups) access Minio through its internal Kubernetes service (`minio-service.shared.svc.cluster.local`).
    -   **External:** For access by users or external tools, it is securely exposed via an Ingress managed by [Traefik](./traefik.md).
-   **Security:** Access to Minio is controlled by access keys and secret keys, which are managed as Kubernetes `Secrets` and injected into the applications that need to access it.

Minio gives us the flexibility and power of cloud object storage, but while keeping the data within our own controlled environment.

### Configuring Webhooks for Event Notifications

Minio can send notifications to a webhook endpoint whenever specific events occur in a bucket (e.g., when an object is created). This is incredibly useful for triggering automated workflows.

The process is divided into two main steps:

#### 1. Define the Webhook in Minio's Configuration

First, you must configure the webhook endpoint in Minio itself. Following Infrastructure as Code (IaC) principles, we do this through environment variables in the Kubernetes `Secret` that holds Minio's configuration.

The variables follow this pattern:
- `MINIO_NOTIFY_WEBHOOK_ENABLE_<ID>`: To enable the webhook.
- `MINIO_NOTIFY_WEBHOOK_ENDPOINT_<ID>`: The URL of the service that will receive the notification.

**Example:**

To create a webhook named `nodered`, you would add the following to the Minio configuration `Secret` (e.g., in `k8s/overlays/my-tenant/namespaces/default/patches/minio-storage-configuration-patch.yaml`):

```yaml
# Inside stringData -> config.env:
export MINIO_NOTIFY_WEBHOOK_ENABLE_NODERED="on"
export MINIO_NOTIFY_WEBHOOK_ENDPOINT_NODERED="http://node-red.security.svc.cluster.local:1880/webhook"
```

When Minio restarts, it registers this endpoint and generates an **ARN (Amazon Resource Name)** for it, which you will need in the next step.

#### 2. Associate the Event with a Bucket

Once the webhook is configured in Minio, you use the `mc` (Minio Client) command-line tool to tell Minio which events on which bucket should trigger a notification to that webhook.

**Installing the Minio Client (`mc`)**

Before you can proceed, you need to install `mc`. You can find the official instructions at the [Minio Client Quickstart Guide](https://min.io/docs/minio/linux/reference/minio-client.html).

For macOS, a common method is to use Homebrew:
```bash
brew install minio/stable/mc
```

**Example:**

To notify the `nodered` webhook every time a file is created (`put`) in the `my-folder/` prefix (folder) of the `my-bucket` bucket, you would run the following command:

```bash
# First, ensure you have an alias set up for your Minio instance.
# If not, create one (this requires port-forwarding the API port, 9000):
# kubectl port-forward svc/minio -n default 9000:9000 &
# mc alias set myminio http://localhost:9000 minio minio123

# Add the notification rule
mc event add --event "put" --prefix "my-folder/" myminio/my-bucket arn:minio:webhook::NODERED
```
- `--event "put"`: Triggers the notification on object creation events.
- `--prefix "my-folder/"`: Filters the events to only those occurring in that "folder".
- `myminio/my-bucket`: The alias and the target bucket.
- `arn:minio:webhook::NODERED`: The ARN of the webhook. Note that the `<ID>` you defined in the environment variable becomes the final part of the ARN.
